# 云原生评估报告

## 1. 云原生技术介绍

  CNCF（Cloud Native Computing Foundation（云原生计算基金会））对云原生定义：云原生技术有利于各组织在公有云、私有云和混合云等新型动态环境中，构建和运行可弹性扩展的应用。云原生的代表技术包括容器、服务网格、微服务、不可变基础设施和声明式API。这些技术能够构建容错性好、易于管理和便于观察的松耦合系统。结合可靠的自动化手段，云原生技术使工程师们能够轻松地对系统作出频繁和可预测的重大变更。

### 1.1 云原生关键技术：容器&不可变基础设施
容器能够将一个包含应用及运行依赖的镜像按照指定资源诉求在物理机或虚机上启动成为一个或一组隔离的OS进程，天然具备敏捷、弹性、可移植的特点。
不可变基础设施（Immutable Infrastructure）是由 Chad Fowler 于2013 年提出的一个很有前瞻性的构想：在这种模式中，任何基础设施的实例（包括服务器、容器等各种软硬件）一旦创建之后便成为一种只读状态，不可对其进行任何更改，如果需要修改需要构建新的实例（服务器、容器），然后替换全部旧的实例。不可变基础设施的优势在于更高的一致性和可靠性，创建、部署、升级的过程是可预测的，给运维带来的好处也很明显：
- (1)	由于每次变更都是使用新版本实例完整的替换旧版本，不依赖当前的运行环境，所以应用实例的升级、回滚更可控。
- (2)	更容易水平扩展。
- (3)	在灾难发生的时候，快速恢复环境。
容器一开始就按照不可变基础设施的理念在设计，容器包括三个基本概念：容器镜像、容器实例、容器仓库。

容器镜像和容器实例一旦创建成功便不能修改，只能替换。容器不可变基础设施的特性使得容器成为云原生时代软件交付的标准模式。

### 1.2 云原生关键技术：声明式API
对于系统的管理有两种方式，一种是声明式一种是命令式。
 
命令式管理指的是运维系统时需要运维人员直接发出运维命令，例如： 运行容器、停止容器等。而声明式管理指的是运维人员在系统中声明最终的状态，而系统根据运维人员声明的状态自动的不断向该状态驱动，最终达到期望的状态。声明式管理的好处是显而易见的，声明式 API 使系统更加健壮，在分布式系统中任何组件随时都可能出现故障，如果所有的故障都需要运维人员介入处理系统的可靠性肯定无法得到保障，此时只有采用声明式管理，系统不需要人工介入即可以自治恢复。
要实现系统的自动声明式管理需要两个条件：通用的状态定义API和智能可扩展的自治处理架构。在云原生技术体系中，实现对容器进行声明式管理和编排的就是Kubernetes。 Kubernetes最主要的设计理念就是声明式API和控制循环（系统自治）。
Kuberentes设计理念—声明式API：
API对象是K8s集群中的管理操作单元。K8s集群系统每支持一项新功能，引入一项新技术，一定会新引入对应的API对象，支持对该功能的管理操作。每个API对象都有3大类属性：元数据metadata、规范spec和状态status：
- 元数据是用来标识API对象的，每个对象都至少有3个元数据：namespace，name和uid。
- 规范描述了用户期望K8s集群中的分布式系统达到的理想状态（Desired State）。
- 状态描述了系统实际当前达到的状态（Status）。   
Kuberentes设计理念—控制循环：
大型分布式系统必然会存在各种异常，比如系统崩溃、容器退出等。Kubernetes 自然会持续关注系统的实时状态，当遇到异常时能够及时的进行自我修复。
- 持续观测、校正，最终将运行状态达到用户期望的状态。
- 感知用户的行为并执行。比如修改 Pod 数量，应用升级/回滚等等。
Kubernetes 实现了大量的 controllers，它们通过 list-watch etcd 来感知集群数据的更新，然后 24 小时不间断的工作以达到期待的状态，在该过程中它们也会把创建的各类数据反馈回 kube-apiserver & etcd，从而形成了数据流的闭环。
Kubernetes 不会基于系统当前状态和期望状态来来确定接下来要执行的一组经过精确计算的相互协调的命令，而是仅基于系统当前状态确定下一个要执行的命令，然后不断迭代，直到没有下一个命令可以执行，系统就达到了稳定状态。
 
Kubernetes架构上由控制面和运行实际业务的数据面（node）组成。
 
Kubernetes 控制面由API Server、Controller、Scheduler 以及 etcd组成。Apiserver作为kubernetes系统的入口，封装了核心对象的增删改查操作，以RESTful接口方式提供给外部客户和内部组件调用。Kubernetes的应用定义如无状态应用Deployment、有状态应用Statefulset等已成为企业抽象不同类型工作负载的的事实标准。Etcd是一个持久的、轻量级的、分布式的、键值数据存储的数据库，它存储了集群的配置信息，并提供了list-watch等高级功能，便于各个控制器很容易获得系统的状态变化，从而快速响应和协调工作。Controller manager 基于控制循环理念负责维护集群的状态，如故障检测、自动扩展、滚动升级等。Scheduler则负资源的调度，按照预定的调度策略将应用调度到相应的节点上。
Kubernetes可扩展性非常强，从一诞生就具备自带平台熟属性：
 
Kubernetes北向提供了容器、集群、服务、应用、配置、租户、网络、存储、策略等等一系列的标准化原子接口，使得多种多样的上层行业场景可以基于自身的应用领域来按需组合使用，并可以通过K8s灵活的CRD（Custom Resource Definitions）机制来面向行业扩展、二次开发相关功能，从而方便地解决企业业务容器化改造、新业务完整技术方案设计、企业流程化方案设计与实施等方面的问题。基于Kubernetes灵活的北向扩展机制，可以将企业内现存的各类基础软件平台，典型的包括以DevOps持续交付、微服务化开发为代表的应用与微服务平台，以AI、大数据、批量计算为代表的数据平台，以及面向5G、物联网、视频等场景的边缘平台，都将会统一到基于Kubernetes的“云原生架构的技术平台”之上。从而，企业基于云原生技术平台将会真正实现对企业内各类业务的统一管理。
 
Kubernetes南向抽象了面向各类运行环境的标准化接口，比如容器运行时的CRI（Container Runtime Interface）、容器网络的CNI（Container Network Interface）、容器存储的CSI（Container Storage Interface），以及各类专用设备与芯片的Device Plugins等，使得企业能够基于自身对性能、安全、高可用、高可靠等关键业务SLA指标的诉求，选取合适的业务运行环境技术体系，将多样化的企业业务运行环境归一到Kubernetes平台统一管理。“应用、平台、设备”三个层面的协同，意味着Kubernetes将真正成为“云原生OS“。

### 1.3 云原生关键技术：微服务
Gartner在2019年12月发表的《将微服务架构原理应用于应用程序架构的解决方案路径》一文中，对微服务和微服务架构有如下定义：“微服务是应用程序组件，它们具有范围狭窄，高度凝聚力，强封装，松散耦合，可独立部署和可独立扩展的特性。它们通过具有定义良好的API合同的开放式标准网络可访问接口作为服务部署。这些组件拥有它们提供的数据，并且仅提供它们拥有的数据。  微服务架构（MSA）是一种设计范例，将SOA和域驱动的设计原理应用于分布式应用程序。MSA具有三个核心目标：开发敏捷性，部署灵活性和精确的可伸缩性。”
软件的最高境界是高内聚、低耦合，符合SOLID原则。但是面对复杂业务系统，采用怎样的方法才能实现系统的高内聚、低耦合呢？微服务就是一种很好的设计理念：微服务试图分离技术实现的复杂性，将复杂的业务逻辑拆解为多个简单逻辑，并通过这些简单逻辑的组合来实现复杂的业务逻辑。通过将庞大、复杂的单体应用分解为合适大小的独立服务的方式来控制业务的复杂性，从而解决软件难以理解、难以演进的难题。从这个意义上讲微服务架构是应用程序的"正确"的下一代体系结构，但微服务不是银弹，为了实现微服务软件的治理需要一台成熟度微服务框架，包括:
 
根据微服务开发模式对开发人员的侵入程度，可以将微服务开发模式分为三代：
第一代服务治理能力内嵌在业务代码中，典型技术包括SOA、ESB等。这种开发模式使用简单，外部依赖少，但是带来了业务代码与服务治理代码代码耦合度高、服务治理代码分散在各个业务模块中，代码重复度高、运维复杂，对开发人员要求也很高。
为了解决第一代开发模式出现的问题，很自然的就出现了以Springcloud、Dubbo为代表的第二代微服务开发模式，即将服务治理能力抽象到统一的SDK实现，业务开发人员基于SDK开发业务代码。这种方式解决了第一代微服务开发代码重复量大、治理逻辑与业务逻辑耦合的问题，但也带来了如下问题：SDK开发语言绑定，无法进行跨语言的统一服务治理，约束了开发语言选型；同时业务逻辑与SDK编译依赖，一旦SDK升级则业务也需要重新编译构建。
 
为了达到服务治理与业务逻辑彻底解耦，将服务治理能力下沉到基础设施的目的，以Istio为代表的服务网格技术应运而生。

### 1.4 云原生关键技术：服务网格
Istio一种云原生的、应用层的、网络技术，用于解决组成应用的组件之间的连接、 安全、策略、可观察性等问题，具有如下特点：
- 多协议（Dubbo、gRPC），多语言（nodejs、.Net、java）兼容遗留老应用成为云原生应用
- 高性能通信，轻量代理。
- 整合容器平台，微服务引擎，API网关，指标监控，日志审计等云上服务，封装为微服务平台，让用户感知不到背后的复杂（服务注册、路由策略、负载均衡、监控数据）
- 可靠的传输复杂网络拓扑中的服务请求，将服务变为现代的云原生服务
从架构上看，Istio是对K8s的完美补充：
- Istio在K8s已有的服务注册、配置、发现基础上增强了服务调用、安全、链路监控、流量治理等微服务关键功能。
- 满足容器应用的部署、伸缩、运维、服务发现等运行诉求。
 
软件架构
 
client端访问backend的所有请求，都通过iptable规则将流量劫持到sidecar容器envoy中处理。这两个访问间的服务发现、熔断、限流、访问控制等均通过Kubernetes CRD的方式在控制面配置。
 
核心理念：
- 非侵入式Sidecar注入技术，将数据面组件注入到应用所在的容器，通过劫持应用流量来进行功能实现，应用无感知。
- 北向API基于K8s CRD实现，完全声明式，标准化。
- 数据面与控制面通过xDS gRPC标准化协议通信，支持订阅模式。
核心特性：
- 服务&流量治理：熔断，故障注入，丰富的负载均衡算法，限流，健康检查，灰度发布，蓝绿部署等。
- 流量与访问可视化：提供应用级别的监控，分布式调用链，访问日志等。
- 安全连接：通过mTLS、认证、鉴权等安全措施帮助企业在零信任的网络中运行应用。
主要应用场景：
- 为应用提供无侵入式流量治理能力。
- 多K8s集群场景提供跨集群流量统一管理能力。




## 2. 云原生设计方法

### 2.1 使用容器交付支撑频繁可控的部署升级
  在云原生架构中，容器交付不仅是应用打包的标准格式，更是实现业务频繁迭代与平滑升级的核心载体。通过构建基于容器的标准化交付流水线，企业能够将应用及其运行环境封装为不可变的镜像制品，彻底消除“开发、测试、生产环境不一致”带来的部署风险，从而支撑起高频次、高可靠性的版本更新。

### 2.2 充分使用云的能力帮助应用实现无状态设计

  软件中服务所有基础组件要求遵循数据与应用分离（服务无状态化），可按需水平扩展；保证在业务量增加，用户增加，数据量增长的情况下能够通过扩容手段保证服务质量。
  软件的业务负载可以通过负载均衡的方式进行分布式的业务负载分发，从而实现服务的弹性伸缩，每个服务具备横向扩展能力，支持按需使用、自动弹性伸缩，可动态替换、灵活部署，支撑高性能、高吞吐量、高并发、高可用业务场景。 
  应用无状态设计实现的难点在于对应用状态的处理，任何状态需要考虑它的分发、处理、存储。应用无状态化的总体思路就是将这些数据存储到后端有状态的中间件中，如分布式数据库、分布式缓存、分布式消息、对象存储、Nosql数据库、文件存储等，这些中间件从设计之初就考虑了自身的弹性、同步等能力，不需要应用层关心。最终系统通过无状态的应用加成熟的中间件实现整体的SLA。
  对于数据的存储，主要分为以下几类数据，应用可以根据建议选择合适的处理方法：
- 内存数据，如Session信息：统一存放到分布式缓存中间件中。
- 应用业务数据：存储到外部数据库中。
- 文件、照片等数据：存储到SFS或对象存储中，并使用CDN预加载。
- 非结构化数：ElasticSearch、mongodb中。
- 应用配置：配置中心，如Kubernetes ConfigMap或Secret。

### 2.3 使用微服务管理业务复杂性

  在传统单体架构中，随着业务规模扩大和功能迭代，代码耦合度会持续升高，任何一个模块的修改都可能引发 “牵一发而动全身” 的风险，不仅开发效率低下，故障排查和系统扩展也变得异常困难 —— 这正是业务复杂性积累到一定阶段的典型痛点。而微服务架构通过 “分而治之” 的核心思想，将复杂的业务系统拆解为一系列独立、可自治的小型服务，每个服务仅聚焦于特定的业务领域（如电商系统中的用户服务、订单服务、支付服务），服务间通过标准化的接口（如 REST、gRPC）进行通信，从根本上解决了单体架构的耦合问题。
  拆分后的微服务具备极强的灵活性，每个服务可以由独立的团队负责开发、测试、部署和维护，团队能够根据业务需求选择最适合的技术栈（如订单服务用 Java 保障稳定性，推荐服务用 Python 提升算法迭代效率），实现 “技术栈解耦”。同时，当某一业务模块需要扩展时（如促销活动期间订单量激增），只需针对性地对订单服务进行水平扩容，无需对整个系统进行升级，既降低了资源浪费，又提升了扩展的精准性。此外，微服务的独立部署特性使得迭代速度大幅提升，单个服务的更新可以在不影响其他服务的前提下完成，缩短了业务功能从开发到上线的周期，让系统能够快速响应市场变化。
  不过，微服务在降低业务复杂性的同时，也引入了分布式系统的管理挑战，如服务注册与发现、负载均衡、分布式事务、链路追踪等问题。因此，在采用微服务管理业务复杂性时，需要配套构建完善的服务治理体系：通过服务注册中心（如 Nacos、Eureka）实现服务的动态发现，借助 API 网关（如 Gateway、Kong）统一管理服务入口，利用分布式事务框架（如 Seata）解决跨服务数据一致性问题，再通过链路追踪工具（如 SkyWalking、Jaeger）实现故障的快速定位。只有将 “拆分” 与 “治理” 相结合，才能真正发挥微服务在管理业务复杂性中的价值，让系统在复杂业务场景下既保持灵活迭代，又具备稳定运行的能力。

### 2.4 面向故障设计，提升业务可靠性

  在云原生环境中，故障是必然发生的常态 —— 服务器硬件损坏、网络链路中断、依赖服务超时、流量突发峰值等问题随时可能出现。传统架构往往以 “规避故障” 为目标，通过采购高可用硬件、优化网络环境等方式降低故障概率，但这种被动防御的思路难以应对云环境的动态不确定性。而 “面向故障设计” 则转变为主动策略，核心是承认故障的必然性，通过提前预设故障场景、构建 “故障韧性”，让系统在故障发生时能够快速恢复，甚至在部分组件故障的情况下仍能保障核心业务的正常运行，最终提升整体业务可靠性 。
  实现面向故障设计的核心手段包括冗余设计、限流熔断与降级、故障注入演练三大方向。冗余设计是基础，通过在多可用区、多地域部署服务实例，确保单一节点或区域故障时，其他节点能立即接管业务 —— 例如，将应用服务器和数据库分别部署在三个不同可用区，当其中一个可用区因断电中断时，负载均衡器会自动将流量转发至正常可用区，实现 “无感知切换”。限流与熔断则聚焦于 “防止故障扩散”：限流通过设定接口的最大并发量（如每秒处理 1000 个请求），避免突发流量冲垮服务；熔断则在服务调用持续失败（如失败率超过 50%）时，暂时切断调用链路，返回预设的兜底响应（如 “服务暂时繁忙，请稍后重试”），防止故障从依赖服务传导至核心服务，保障系统 “不雪崩”。
  故障注入演练则是验证系统韧性的关键环节。通过主动向系统注入故障（如模拟服务器宕机、网络延迟、数据库读写失败等），观察系统的响应和恢复过程，发现潜在的设计缺陷 —— 例如，电商平台在大促前通过故障注入工具关停 30% 的订单服务实例，测试剩余实例是否能通过负载均衡承接流量，以及降级策略是否能保障用户 “下单” 核心功能。此外，完善的监控与自愈能力也是面向故障设计的重要支撑：通过全链路监控工具实时采集服务的 CPU 使用率、响应时间、错误率等指标，一旦发现异常（如错误率突升），自动触发告警并执行自愈操作（如重启服务、扩容实例），将故障对业务的影响降至最低。

### 2.5 通过 CI/CD 实施DevOps 和自动化
  DevOps 的核心目标是打破开发（Dev）、测试（Test）、运维（Ops）团队之间的壁垒，实现 “从代码提交到业务上线” 的高效协作与流程自动化，而 CI/CD（持续集成 / 持续交付 / 持续部署）正是落地 DevOps 的核心工具链与流程载体。CI（持续集成）聚焦于 “开发阶段的协作与质量保障”，要求开发人员频繁将代码提交至共享代码仓库（如 GitLab、GitHub），通过自动化工具（如 Jenkins、GitLab CI）触发构建、编译、单元测试、代码质量扫描（如 SonarQube）等流程，确保每一次代码提交都能快速验证可行性 —— 例如，当开发人员提交一段订单模块的代码后，CI 流程会自动编译代码，运行预设的单元测试用例，若测试失败或代码存在漏洞，会立即向团队反馈，避免问题代码进入后续环节，从源头减少集成风险。
  CD 则分为持续交付（Continuous Delivery）和持续部署（Continuous Deployment），前者聚焦于 “将代码安全、快速地交付至生产环境就绪状态”，后者则实现 “交付后的自动化部署”。在持续交付阶段，经过 CI 验证的代码会被自动打包为镜像（如 Docker 镜像），推送至镜像仓库（如 Harbor），随后触发自动化集成测试、性能测试（如 JMeter）、安全扫描（如 Trivy）等流程，验证镜像在模拟生产环境中的可用性；所有测试通过后，镜像会被标记为 “可部署状态”，等待人工审批后即可部署至生产环境。而持续部署则在此基础上进一步自动化，当镜像通过所有验证后，无需人工干预，通过部署工具（如 ArgoCD、Flux）自动将镜像部署至生产环境的 Kubernetes 集群，实现 “代码提交后，几小时内完成上线” 的高效迭代。
  通过 CI/CD 实施 DevOps 和自动化，不仅能大幅提升研发效率，更能保障业务的稳定性与可追溯性。一方面，自动化流程替代了传统的人工编译、测试、部署等重复性工作，将研发团队从繁琐的操作中解放出来，专注于业务逻辑开发，同时避免了人工操作可能出现的失误（如部署时漏传配置文件）。另一方面，CI/CD 流程通过日志记录每一个环节的操作（如谁提交了代码、哪次测试失败、何时完成部署），实现了全流程可追溯，一旦生产环境出现问题，能够快速定位到具体的代码版本和部署环节，缩短故障排查时间。此外，自动化的质量与安全检测，让代码在进入生产环境前就经过多轮验证，降低了线上故障的概率，最终实现 “快速迭代” 与 “稳定运行” 的平衡，这正是云原生环境下业务持续发展的核心需求。
